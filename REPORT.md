# Secure Coding 과제 보고서 — Tiny Secondhand Shopping Platform
https://github.com/xiaohuasheng105/secure-coding
## 0. 개요

- 과제: 중고거래 플랫폼 개발 및 시큐어 코딩 적용
- 개발 방법론: Waterfall 5단계(요구사항 도출 → 시스템 설계 → 시스템 구현 → 테스트 → 유지보수)를 그대로 따르되,
  구현 중 발견된 보안 약점은 즉시 설계 단계로 되돌아가 반영하는 반복적 방식으로 진행했다.

## 1. 요구사항 도출

### 1.1 요구사항 (필수)

| 기능 | 비고 |
|---|---|
| 회원가입 / 로그인 | 아이디 중복 불가, DB로 관리 |
| 상품 등록 / 조회 | 등록/관리는 본인만, 조회는 전체 공개 |
| 사용자 간 소통 | 전체 채팅 + 1:1 채팅 |
| 악성 유저/상품 차단 | 신고 기반, 누적 시 자동 차단·휴면 |
| 유저 간 송금 | 신규 설계 필요 항목 |
| 상품 검색 | 신규 설계 필요 항목 |
| 관리자 기능 | 신규 설계 필요 항목 |

### 1.2 비기능 요구사항

- 보안: 체크리스트(`secure_coding_checklist_result.csv`) 28개 항목 전체 충족
- 유지보수성: 단일 파일에 몰려 있던 기존 코드(`app.py` 1개)를 관심사별로 분리(`db.py`, `security.py`)
- 사용성: 최소한의 서버 사이드 렌더링 UI로 모든 기능에 접근 가능해야 함

### 1.3 스타터 코드 분석 결과

기존 스타터 코드(`app.py`, 210줄)를 분석한 결과 다음 기능이 이미 구현되어 있었다:
회원가입, 로그인/로그아웃, 대시보드(상품 목록 + 전체 채팅), 프로필(소개글 수정), 상품 등록/상세조회, 신고 접수(저장만).

반면 다음은 설계상 요구되었으나 구현되어 있지 않았다:
사용자 조회, 등록 상품 관리(수정/삭제), 1:1 채팅, 신고 누적에 따른 자동 차단/휴면.
송금, 검색, 관리자 기능은 애초에 설계 자체가 없었다.

## 2. 시스템 설계

### 2.1 기능 목록 (최종)

- **유저 관리**: 회원가입, 로그인(실패 잠금 포함), 로그아웃, 사용자 조회(타인 프로필), 마이페이지(소개글/비밀번호 변경)
- **상품 관리**: 등록, 목록/검색, 상세조회, 수정/삭제(소유자 한정), 내 상품 관리
- **소통**: 전체 실시간 채팅(Socket.IO), 1:1 채팅(방 단위, 이력 영구 저장)
- **악성 유저 필터링**: 신고 접수(사용자/상품 대상), 중복 신고 방지, 시간당 신고 횟수 제한, 누적 신고 시 자동 차단/휴면
- **송금**: 회원가입 시 모의 잔액 지급, 사용자명 기반 송금, 비밀번호 재인증, 거래 내역 조회
- **관리자**: 대시보드(통계), 사용자 휴면/해제, 상품 차단/해제/삭제, 신고 내역 조회

### 2.2 웹페이지 설계

기본(index) / 회원가입 / 로그인 / 대시보드(+검색+전체채팅) / 상품등록 / 상품상세 / 상품수정 / 내 상품관리 /
프로필 / 타인 프로필 / 신고 / 송금 / 쪽지(1:1채팅) 목록 / 1:1채팅방 / 관리자(대시보드·사용자·상품·신고) / 에러(403·404·500)

### 2.3 데이터베이스 설계

```
user(id, username UNIQUE, password_hash, bio, balance, is_admin, is_banned,
     failed_login_count, lock_until, created_at)
product(id, title, description, price, seller_id, status['active'|'blocked'], created_at)
report(id, reporter_id, target_type['user'|'product'], target_id, reason, created_at,
       UNIQUE(reporter_id, target_type, target_id))
message(id, room, sender_id, content, created_at)
"transaction"(id, sender_id, receiver_id, amount, created_at)
```

기존 설계 대비 `password` → `password_hash`, `balance/is_admin/is_banned/failed_login_count/lock_until`,
`product.status`, `message`, `transaction` 테이블을 신규/변경했다. 근거는 3장의 보안 조치 내역 참고.

### 2.4 임계치/정책 설계값

- 로그인 실패 5회 → 5분 계정 잠금
- 상품 신고 3회 누적 → 자동 차단(비활성화, 관리자가 해제 가능)
- 사용자 신고 5회 누적 → 자동 휴면(로그인 불가, 관리자가 해제 가능)
- 신고는 사용자당 시간당 10회로 제한, 동일 대상 중복 신고 불가
- 채팅 메시지는 10초당 5회로 제한, 500자 초과 잘림
- 세션은 30분 유휴 시 만료

## 3. 시스템 구현 및 보안 약점 조치 내역

구현은 `app.py`(라우트/Socket.IO), `db.py`(스키마), `security.py`(인증/검증/레이트리밋)로 분리했다.
아래는 스타터 코드에서 실제로 발견한 보안 약점과 조치 내용이다.

| # | 발견한 문제 (스타터 코드) | 조치 내용 |
|---|---|---|
| 1 | 비밀번호를 `INSERT INTO user (... password ...)`로 **평문 저장** | Argon2(`argon2-cffi`)로 해시 후 저장(`security.hash_password`), 검증은 `verify_password` |
| 2 | `SECRET_KEY = 'secret!'` **하드코딩** | 환경변수 `SECRET_KEY`로 주입, 미설정 시 프로세스별 랜덤 생성 |
| 3 | CSRF 토큰 없이 모든 폼이 세션 쿠키만으로 상태 변경 요청 처리 | `Flask-WTF CSRFProtect` 전역 적용 + 전 POST 폼에 `csrf_token` 히든 필드 |
| 4 | 회원가입/로그인 시 사용자명·비밀번호 **서버측 검증 전무** (빈 문자열도 가입 가능) | 정규식 기반 검증(`USERNAME_RE`, `PASSWORD_RE`), 미충족 시 가입/변경 거부 |
| 5 | 로그인 실패 횟수 제한 없음 → 무제한 비밀번호 대입 공격 가능 | 5회 실패 시 5분 계정 잠금(`user.failed_login_count`, `lock_until`) |
| 6 | 상품/신고 입력값에 길이 제한 없음, 제어문자 그대로 저장 | `clean_text()`로 제어문자 제거 및 필드별 길이 제한(제목 100자, 설명 2000자, 신고사유 500자 등) |
| 7 | 상품 수정/삭제 기능 자체가 없어 **소유자 검증 로직 부재** | 신규 구현 시 `product.seller_id == 현재 사용자 id` 확인 후 불일치 시 403 |
| 8 | 신고는 저장만 되고 아무 조치가 없어 악성 유저/상품이 방치됨 | 상품 3회/사용자 5회 누적 시 자동 차단·휴면 처리(`_apply_report_threshold`) |
| 9 | 동일 사용자가 동일 대상을 무한정 반복 신고 가능(신고 남용) | `report` 테이블에 `UNIQUE(reporter_id, target_type, target_id)` 제약 + 시간당 10회 제한 |
| 10 | Socket.IO `send_message`에 **로그인 여부 확인이 없어** 비로그인 사용자도 메시지 송신 가능, `username`도 클라이언트가 임의로 지정 가능(스푸핑) | 서버가 `session['user_id']`로 사용자 조회 후 `username`을 서버측 값으로 강제 설정, 미인증/휴면 계정은 무시 |
| 11 | 채팅에 메시지 개수 제한이 없어 스팸/DoS성 남용 가능 | `RateLimiter`로 10초당 5회 제한 |
| 12 | Socket.IO 클라이언트 스크립트를 외부 CDN(cdnjs)에서 로드 → CDN 변조/장애 시 전체 서비스에 영향, CSP 설정도 어려움 | `static/js/socket.io.min.js`로 로컬 vendoring 후 `script-src 'self'` CSP 적용 |
| 13 | 응답에 보안 헤더 전무(CSP, X-Frame-Options, X-Content-Type-Options 등 없음) | `after_request`에서 CSP/X-Frame-Options/X-Content-Type-Options/Referrer-Policy 설정 |
| 14 | `debug=True` 상태로 운영 시 스택 트레이스·소스코드가 그대로 사용자에게 노출 | `FLASK_DEBUG` 환경변수 기반으로 기본값 `False`, 커스텀 403/404/500 에러 페이지로 대체, 예외는 서버 로그(`app.logger`)에만 기록 |
| 15 | 세션 쿠키에 `HttpOnly`/`SameSite` 등 명시적 설정 없음(Flask 기본값 의존) | `SESSION_COOKIE_HTTPONLY=True`, `SESSION_COOKIE_SAMESITE='Lax'`, `FORCE_HTTPS=1` 환경변수로 `Secure`+HSTS 활성화 |
| 16 | 비밀번호 변경 기능 자체가 없었고, 신규 설계한 송금 기능도 세션만으로 처리하면 세션 탈취 시 자금 이동 위험 | 비밀번호 변경·송금 등 민감 작업에 **현재 비밀번호 재인증** 요구 |
| 17 | 송금 로직을 단순 SELECT 후 UPDATE로 구현할 경우 동시 요청 시 잔액 이중 차감(TOCTOU) 가능 | `BEGIN IMMEDIATE`로 트랜잭션 시작 시 즉시 쓰기 락을 획득하여 동시 송금 요청을 직렬화, 실패 시 `rollback` |
| 18 | 상품 검색을 `LIKE '%' + q + '%'`로 구현할 경우 `%`, `_`를 사용자가 넣어 의도치 않은 결과를 조회하는 LIKE 와일드카드 인젝션 가능 | `LIKE ... ESCAPE '\\'` 절과 함께 입력값의 `%`, `_`, `\`를 이스케이프 처리 |
| 19 | 관리자 권한을 부여하는 기능이 없으면 이후 신규 도입 시 누구나 웹에서 자기 자신을 관리자로 승격시키는 취약점이 생기기 쉬움 | 관리자 승격은 **웹 라우트로 노출하지 않고** `flask create-admin` CLI 명령으로만 가능하도록 분리 |
| 20 | SQL 실행 시 이미 파라미터 바인딩(`?`)을 사용 중이었으나, 신규 추가 코드에서도 문자열 포매팅으로 SQL을 조립할 위험이 있음 | 신규 코드 전체(검색/송금/신고/관리자 기능 포함)에서 예외 없이 `?` 파라미터 바인딩만 사용 |

## 4. 체크리스트 작성 및 테스트

체크리스트는 강의자료를 기준으로 각 항목별 조치 여부와 근거 코드를
`secure_coding_checklist_result.csv`에 정리했다. 28개 항목 중 26개 완료, 2개는 "부분 완료"(TLS 종단, DB 계정 권한
분리 — SQLite 특성상 해당 없음/리버스프록시 구성 필요)로 표시했다.

### 4.1 실제 테스트 (수행 및 결과)

로컬에서 서버를 구동한 뒤 `curl`/`python-socketio` 클라이언트로 아래 시나리오를 검증했다.

| 시나리오 | 기대 결과 | 실제 결과 |
|---|---|---|
| 회원가입 → 로그인 → 대시보드 접근 | 정상 처리 | 통과 |
| 상품 등록 후 검색(`/search?q=...`) | 등록한 상품이 검색됨 | 통과 |
| 다른 사용자가 타인 상품 수정 페이지 접근 | 403 | 통과 |
| 다른 사용자가 CSRF 토큰을 포함해 타인 상품 삭제 시도 | 403(소유자 아님) | 통과 |
| CSRF 토큰 없이 상태 변경 요청 | 400(CSRF 오류) | 통과 |
| 동일 사용자를 5명이 각각 신고 | 5번째 신고 시 자동 휴면 처리 | 통과 |
| 휴면 처리된 계정으로 로그인 시도 | 로그인 거부 안내 | 통과 |
| 관리자 계정이 휴면 해제 | 즉시 재로그인 가능 | 통과 |
| 동일 계정으로 5회 연속 비밀번호 오류 후 정상 비밀번호로 로그인 | 잠금 안내(로그인 거부) | 통과 |
| 송금 시 비밀번호 오입력 | 송금 거부 | 통과 |
| 송금 성공 후 양측 잔액 확인 | 송금자 -금액 / 수신자 +금액 | 통과 |
| 비로그인 상태로 Socket.IO에 접속해 채팅 전송 | 서버가 메시지를 무시(다른 클라이언트에 전달 안 됨) | 통과 |
| 로그인 사용자가 10초 내 8개 메시지 전송 | 5개까지만 수신됨(레이트리밋) | 통과 |
| 응답 헤더 확인 | CSP/X-Frame-Options/X-Content-Type-Options 존재 | 통과 |

## 5. 유지보수

- 실제 사용해본 결과, 신고 화면에서 신고 대상 ID를 직접 입력해야 하는 점이 불편하여 상품 상세페이지/타인
  프로필 페이지에 "신고하기" 링크를 추가하고 대상 종류·ID를 자동으로 채워주도록 개선했다
- 1:1 채팅 진입 시 상대를 지정할 방법이 없어 "쪽지" 메뉴(대화 상대 목록)와 타인 프로필의 "1:1 채팅하기" 링크를
  추가로 설계했다.
- 향후 개선 과제: (1) 다중 프로세스/서버 환경에서는 현재의 인메모리 rate limiter를 Redis 등
  외부 저장소 기반으로 교체, (2) SQLite는 소규모 트래픽에는 충분하나 동시 접속자가 늘어나면 PostgreSQL
  등으로 이전을 고려

## 6. 결론

스타터 코드는 핵심 기능의 골격은 갖추었으나 인증/인가, 입력 검증, 신고 처리, 실시간 통신 보안이
사실상 구현되어 있지 않았다. 본 과제에서는 요구사항에 명시된 미구현 기능(상품관리, 1:1채팅, 신고누적차단)과
신규 설계 기능(송금, 검색, 관리자)을 모두 구현하면서, 체크리스트 28개 항목을 기준으로 총 20건의 구체적인
보안 약점을 식별하고 조치했다. 특히 "기능 구현에 매몰되어 보안을 놓치기 쉽다"는 강의 취지에 맞추어, 신규 기능을
추가할 때마다(송금의 동시성 문제, 검색의 LIKE 인젝션, 채팅의 인증 부재 등) 그 기능 고유의 보안 위협을 함께
분석하고 설계에 반영하려고 했다.
